How to Learn Go (Without Losing Your Mind)

Before you write a single line of code, let's talk about mindset, tools, and how to not rage-quit.

The Python → Go Mindset Shift

Coming from Python, Go will feel annoyingly strict at first. This is actually a feature.

"Go is designed to make you think before you type, not debug after you run."

Things That Will Annoy You (And Why They're Good)

  • Unused variables are errors: Forces clean code. No x = 5 sitting around doing nothing.
  • Explicit error handling: No hidden exceptions. You know exactly what can fail.
  • No generics... wait, now there are: Go 1.18+ has generics, but the culture prefers simplicity.
  • Curly braces on the same line: Not negotiable. gofmt will enforce this.
  • No classes: Structs + interfaces = more flexible, less OOP baggage.
Key Insight

Go's restrictions aren't limitations — they're guardrails. The language is designed to make the "obvious" code also the "correct" code.

Setting Up Your Environment

Install Go

Installation
# macOS
brew install go

# Ubuntu/Debian
sudo apt install golang-go

# Or download from https://go.dev/dl/

# Verify
go version
# go version go1.21.0 darwin/arm64

Editor Setup

Use VS Code + Go extension or GoLand. The tooling is incredible:

  • gopls: Language server — autocomplete, go-to-definition, refactoring
  • gofmt: Auto-formats on save (no more style arguments)
  • go vet: Catches suspicious code patterns
  • staticcheck: Even more linting
VS Code settings.json
{
    "[go]": {
        "editor.formatOnSave": true,
        "editor.defaultFormatter": "golang.go"
    },
    "go.lintTool": "staticcheck",
    "go.lintOnSave": "package"
}

Your First Project

Creating a module
# Create project directory
mkdir myproject && cd myproject

# Initialize module (like pip's setup.py or package.json)
go mod init github.com/yourusername/myproject

# This creates go.mod — your dependency file
cat go.mod
# module github.com/yourusername/myproject
# go 1.21

How to Read Go Docs (Actually Read Them)

Go has incredible documentation. Learn to use it instead of blindly copying StackOverflow.

The Go Documentation Sites

  • pkg.go.dev — Package documentation (like PyPI + readthedocs combined)
  • go.dev/doc — Official guides, tutorials, specs
  • go.dev/blog — Deep dives on language features

Reading Package Docs

Let's look at the strings package:

Finding docs
# From terminal
go doc strings
go doc strings.Split

# Or visit https://pkg.go.dev/strings

A typical function doc looks like:

From pkg.go.dev/strings
func Split(s, sep string) []string

// Split slices s into all substrings separated by sep and returns
// a slice of the substrings between those separators.
//
// If s does not contain sep and sep is not empty, Split returns a
// slice of length 1 whose only element is s.
//
// If sep is empty, Split splits after each UTF-8 sequence.
// If both s and sep are empty, Split returns an empty slice.
Pro Tip

Scroll down on pkg.go.dev to see Examples. Most stdlib packages have runnable examples that show real usage.

The Source Code Is Your Friend

Go's stdlib is readable. When in doubt, read the source:

Reading source
# Find where a package lives
go list -f '{{.Dir}}' strings
# /usr/local/go/src/strings

# Or click "Source" on pkg.go.dev

When the Compiler Yells at You

Go's compiler errors are actually helpful once you learn to read them. Let's decode the common ones.

undefined: someVar

You're using a variable that doesn't exist in this scope. Check spelling, or maybe you declared it in an if-block.

someVar declared but not used

Delete it or use it. If you intentionally don't need a return value, use _.

cannot use x (type Y) as type Z

Type mismatch. Go is strict. You probably need a conversion: int(myFloat) or string(myBytes).

not enough arguments in call to someFunc

Check the function signature. You're missing parameters.

multiple-value someFunc() in single-value context

The function returns multiple values (probably result, err). You need to handle both: result, err := someFunc()

The Debug Loop

  1. Read the error: Line number, what it says
  2. Check the line: Is it a typo? Type mismatch?
  3. Check the function signature: go doc package.Function
  4. If stuck: Search "golang [error message]" — someone's had this before

How to Approach Problems

When you're stuck, here's a framework that works:

The "5 Minute Rule"

  1. 0-2 min: Read the error. Actually read it.
  2. 2-4 min: Check docs for the function/type involved
  3. 4-5 min: Google it / ask ChatGPT
  4. 5+ min: Isolate the problem — make a minimal example that reproduces it

When Learning New Concepts

  1. Read the explanation
  2. Type the example yourself (no copy-paste!)
  3. Break it — what happens if you remove this line?
  4. Modify it — can you make it do something slightly different?
  5. Build something small using the concept
Don't Do This

Copy-pasting code without understanding it. You'll learn nothing and have no idea how to fix it when it breaks.

Rubber Duck Debugging

Explain your code out loud, line by line. To a duck, a friend, or just yourself. You'll often realize the bug while explaining.

Tools You'll Use Daily

Go commands cheat sheet
# Run code directly
go run main.go
go run .                    # Run all files in current dir

# Build binary
go build -o myapp .

# Run tests
go test ./...              # All tests in all subdirs
go test -v ./...           # Verbose
go test -run TestName      # Run specific test

# Manage dependencies
go get github.com/some/package    # Add dependency
go mod tidy                       # Clean up go.mod
go mod download                   # Download all deps

# Code quality
go fmt ./...               # Format all code
go vet ./...               # Find suspicious code
staticcheck ./...          # Advanced linting

# Documentation
go doc strings             # Package docs
go doc strings.Split       # Function docs
go doc -all strings        # Everything

Getting Unstuck

Good Places to Ask

  • r/golang — Active, helpful community
  • Gophers Slack — Real-time help
  • StackOverflow — [go] tag, for specific technical questions

How to Ask Good Questions

  1. What are you trying to do? (Goal, not implementation)
  2. What did you try? (Show code)
  3. What happened? (Full error message)
  4. What did you expect?
The XY Problem

Don't ask "How do I do X?" when your real problem is Y. Explain the actual goal.

Module 0 Summary

  • Mindset: Go's strictness is a feature, not a bug
  • Docs: pkg.go.dev is your primary resource. Learn to read examples.
  • Errors: Read them. They tell you exactly what's wrong.
  • Learning: Type code yourself, break it, modify it, build with it
  • Tools: go run, go test, go doc, go fmt
